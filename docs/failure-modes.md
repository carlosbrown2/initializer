# Failure-mode register

Every module, function, and data flow that can fail must appear here, paired with a mechanical check that catches the failure before merge. See `project-kickoff-prompt.md` §1 for the contract.

**Status values:** `covered` | `proven-impossible` | `out-of-scope`

**Categories:** `correctness` | `concurrency` | `atomicity` | `input` | `resource` | `temporal` | `version` | `dependency` | `operational` | `security`

| CLAUDE.md verification gate command | soft-fail escape in a correctness chain — trailing `\|\| <always-succeed>` (`\|\| true`, `\|\| :`, `\|\| 0`, `\|\| exit 0`, etc.) binds to the whole `&&` chain so any in-chain failure silently exits 0; or a non-bash file (markdown) is added to a `bash -n` chain forcing a soft-fail escape. The pre-push hook re-runs whatever the gate is, so a no-op gate makes the hook a no-op too. The bug class name is broader than the original `\|\| true` label because `\|\|` precedence makes every `<always-succeed>` trailer structurally identical | correctness | tests/hooks/gate.bats asserts the extracted gate command has no soft-fail-escape tail (via scripts/hooks/parsers.sh gate_has_soft_fail_escape which covers `\|\| true`, `\|\| :`, `\|\| 0`, `\|\| exit 0`), contains no `.md` paths, passes against the current repo, and propagates non-zero exit when any in-chain clause is corrupted | covered |

| scripts/hooks/parsers.sh syntax errors | a syntax error in parsers.sh silently breaks every register-integrity check (install.sh sources it at runtime); without a gate clause, the failure surfaces only when a developer hits pre-commit on an affected register edit | correctness | CLAUDE.md verification gate includes `bash -n scripts/hooks/parsers.sh`; tests/hooks/gate.bats "gate fails when scripts/hooks/parsers.sh has a syntax error" corrupts parsers.sh in a temp tree and asserts the chain exits non-zero, which also fail-closes a future accidental drop of the parsers.sh clause from the gate | covered |
| scripts/hooks/install.sh review artifact validator | validator only matched clause-shape (`P[123]\.[a-z][a-z-]*`) and did not check membership in the rubric, so a review artifact citing `P1.totally-made-up-clause` passed exit 0 — turning the "Review verdict" decision-register row into a Goodhart on clause-shape rather than on the actual canonical clause set | correctness | scripts/hooks/parsers.sh review_artifact_clauses_check extracts the canonical clause set from `**P[123].name**` bold-marker definitions in docs/skills/review-rubric.md and rejects any cited clause not in that set; scripts/hooks/install.sh wires it into the review-artifact validator so review-bead commits with invented clauses block; tests/hooks/parsers.bats covers reject (invented clause), accept (real clauses), accept-after-rubric-add (no separate registration), reject-when-rubric-empty, plain-prose-citation reject, and a smoke test that synthesizes an artifact citing every real clause from the shipped rubric | covered |
| CLAUDE.md "## Verification Gate" extractor duplication | the awk that extracts the gate command was duplicated in scripts/hooks/install.sh (pre-push hook heredoc) and tests/hooks/gate.bats (setup). Drift between the two makes the gate.bats "gate passes against the current repo" sanity test a proxy for "what gate.bats extracts," not "what the pre-push hook extracts" — so a gate.bats extractor that silently extracts nothing would make every gate assertion vacuously pass while the real pre-push hook happily re-runs a drifted gate | correctness | scripts/hooks/parsers.sh gate_command_extract is the single implementation; the pre-push heredoc in scripts/hooks/install.sh sources parsers.sh and calls the function; tests/hooks/gate.bats sources parsers.sh in setup; tests/hooks/gate.bats also asserts (a) install.sh's pre-push heredoc uses gate_command_extract, (b) neither install.sh nor gate.bats re-inlines the awk anchor pattern, (c) end-to-end equivalence: the heredoc-extracted pre-push body's GATE_CMD matches gate_command_extract's output byte-for-byte against the same CLAUDE.md | covered |
| scripts/ralph/ralph.sh routing logic | confidence parsing, auto-land policy matrix, retry-tracking transitions, and REWORK prerequisite extraction had no mechanical coverage — only `bash -n scripts/ralph/ralph.sh`. Two latent bugs survived: `read_auto_land_policy` used `grep -A1 "## Confidence Routing"` which misses the auto-land line across a blank line (silently returns the default regardless of the real setting), and the REWORK prereq regex with `head -1` returned the active bead id from the "X depends on:" header line instead of the real prerequisite (re-open step was a no-op, review bead looped) | correctness | scripts/ralph/lib.sh extracts parse_confidence / read_auto_land_policy / should_auto_land / compute_retry_state / extract_prereq_bead_id as a sourceable library; scripts/ralph/ralph.sh sources it at startup (fails closed if absent); tests/hooks/ralph.bats exercises each function with the bead-flagged adversarial cases (placeholder must not match, blank line between heading and value, commented-out alternative, unknown policy fallback, MAX_RETRIES boundary, active-bead exclusion in dep-list output); the bats suite already runs under `bats tests/hooks/` in the CLAUDE.md verification gate | covered |
| scripts/ralph/ralph.sh ACTIVE_BEAD snapshot | the active bead id was captured twice per iteration — once before the agent ran (as CURRENT_BEAD for the "Current bead:" display) and again after (as ACTIVE_BEAD for signal routing and confidence log). On successful BEAD_DONE iterations the agent had already called `bd close` between the two captures, so the second read returned empty and every auto-land log line recorded `bead=unknown`. The signature is visible in confidence.log: iter=1..12 all `bead_done=true bead=unknown`, interspersed with iter=2 and iter=8 showing real ids because the agent stalled without closing. This made the confidence log useless for per-bead auditing | correctness | scripts/ralph/ralph.sh captures `_RALPH_ACTIVE_BEAD` exactly once at the top of each iteration (replacing both the CURRENT_BEAD and post-run ACTIVE_BEAD reads); the id is reused for display, signal routing, retry tracking, and confidence logging. The snapshot-volatile-state-once pattern is promoted to CLAUDE.md Discovered Patterns so future external-state reads around agent invocations inherit the rule | covered |
| scripts/hooks/install.sh bead-type gate, bd extraction | the pre-commit hook conditioned on `IN_PROGRESS_BEAD` which was populated by `bd list | grep -o ... | head -1 || true`. If bd's human-readable format drifted, the grep matched nothing and the entire gate chain (bead-type, rubric-edit, scope, review artifact) silently no-opped. A malformed bd output was indistinguishable from "no bead in progress," which is exactly the state that disables the chain | correctness | scripts/hooks/parsers.sh bd_bead_in_progress uses `bd list --status=in_progress --json \| jq` and returns non-zero on extraction failure (bd errored or produced non-parseable JSON). scripts/hooks/install.sh pre-commit sources the function and BLOCKS the commit on non-zero — fail-closed rather than fail-open. Phase 1 bootstrap (no bd installed) still passes because the function short-circuits on `command -v bd`. tests/hooks/parsers.bats covers the non-parseable-JSON reject path and the no-bd-installed accept path | covered |
| scripts/hooks/install.sh commit-msg format | the commit-msg hook only validated the `<type>:` prefix. A commit of `feat: banana` passed even though prompt.md advertises the `<type>: [bead-id] - <title>` format for bead commits. The hook was theater — the format was a prompt-level suggestion with no structural backstop, so "commit message format" was `ritual-bounded` dressed as `bounded` | correctness | scripts/hooks/install.sh commit-msg hook now enforces the bracketed `[bead-id] - <title>` shape when the message starts with `[`, using a regex sourced from BEAD_ID_REGEX. Non-bead commits (no `[` after the type prefix) still pass if the description is non-empty. tests/hooks/parsers.bats covers the accept / reject matrix for both forms | covered |
| scripts/ralph/archive.txt machine-readability | prompt.md mandates the agent append a `## <date> - <bead-id>` block to archive.txt after each BEAD_DONE iteration, but nothing downstream parsed it. "Agent updated the log" was a proxy for "the log is useful"; the agent could write anything (or nothing structured) and the gate stayed green. Future agents discovering prior decisions had to trust that each block was present, complete, and findable | correctness | scripts/hooks/parsers.sh archive_schema_check asserts that every `bead_done=true` entry in confidence.log with a real bead id (not `unknown`) has a matching `## YYYY-MM-DD [HH:MM] - <bead-id>` block in archive.txt. Wired into the verification gate so a BEAD_DONE without a corresponding archive block fails the next push. Phase 1 bootstrap (no confidence.log) and pre-fix logs (all `bead=unknown`) pass. Smoke tested in tests/hooks/parsers.bats | covered |
| scripts/ralph/lib.sh vs scripts/hooks/parsers.sh bead id regex | bead id regex was inlined in two libraries with drifted values (lib.sh used `[a-z][-a-z0-9]*-[a-z0-9]{2,}`, install.sh grep used `[a-z_]*-[a-z0-9]*-[a-z0-9]*`). Each was individually plausible; together they disagreed about what counts as a bead id. install.sh could see a bead the REWORK prereq extractor couldn't, or vice versa. The drift was silent because no test compared the two | correctness | scripts/ralph/lib.sh `BEAD_ID_REGEX` and scripts/hooks/parsers.sh `PARSERS_BEAD_ID_REGEX` are the single-source constants each library reads. A bd downgrade or a bd format change predating 0.3 surfaces as a gate failure on every pre-push and on every CI run | covered |
